## Validation patterns

Use Zod for all external input (form data, Route Handler bodies, URL params).

```typescript
// src/types/transaction.ts
import { z } from 'zod'

export const transactionSchema = z.object({
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  type: z.enum(['expense', 'income', 'transfer', 'adjustment', 'opening_balance']),
  account_id: z.string().uuid(),
  amount: z.number().multipleOf(0.01).positive(),
  category_id: z.string().uuid().nullable().optional(),
  payee: z.string().max(255).optional(),
  memo: z.string().max(1000).optional(),
  tags: z.array(z.string().max(50)).default([]),
  next_month: z.boolean().default(false),
})

export type TransactionInput = z.infer<typeof transactionSchema>
```

**Rules:**
- Never trust client-supplied `user_id` — always derive from `supabase.auth.getUser()` server-side.
- Validate at the Route Handler boundary; internal functions may assume valid data.
- All monetary amounts are `NUMERIC(12,2)` in DB — enforce 2 decimal places, reject more.

## Error handling

Typed Result pattern for all server operations:

```typescript
type Result<T> = { data: T; error: null } | { data: null; error: string }
```

**Route Handlers:**
```typescript
// success
return NextResponse.json({ data: result })

// failure
return NextResponse.json({ error: 'Message' }, { status: 400 })  // validation
return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
return NextResponse.json({ error: 'Internal error' }, { status: 500 })
```

**Client Components:**
- Surface errors via shadcn/ui `Sonner` toast.
- Never swallow errors silently — always `console.error` in development.
- User-facing messages must be human-readable, not raw Supabase error strings.

## Auth patterns

```typescript
// Server Component or Route Handler — verify session
import { createServerClient } from '@/lib/supabase/server'

const supabase = createServerClient()
const { data: { user } } = await supabase.auth.getUser()
if (!user) redirect('/login')
// user.id is the safe, server-derived user_id to use in all DB writes
```

```typescript
// src/middleware.ts — protects all /(app)/* routes
// Checks session cookie, redirects unauthenticated users to /login
// Also checks email_confirmed_at — unverified users redirected to /verify-email
```

```typescript
// Client Component — auth state subscription (root layout)
import { createBrowserClient } from '@/lib/supabase/client'
const supabase = createBrowserClient()
supabase.auth.onAuthStateChange((event, session) => {
  if (event === 'SIGNED_OUT') router.push('/login')
})
```

**Inactivity timeout:** 30-minute timer in `useInactivityTimer` hook.
On timeout: call `supabase.auth.signOut()` then `router.push('/login')`.

**"Recuérdame":** Pass `{ persistSession: true }` to `signInWithPassword()`.
Without it, session ends when the browser tab closes (default Supabase behavior with cookies is persistent; must set session duration to 0 for non-persistent).

**Token storage:** Access token in memory only. Refresh token in `httpOnly; Secure; SameSite=Strict` cookie — managed by `@supabase/ssr`, do not override.

| 2026-07-09 | M03 | ~~"Disponible para ahorrar/invertir" reuses the global "Dinero a Asignar", not a per-account balance~~ — **superseded 2026-07-10, see below** | `budget_allocations` has no `account_id` — the budget engine (M08) is computed globally across all on-budget accounts, so there is no way to derive a true per-account "free to save" figure without a schema redesign. `/accounts` composes `/api/accounts` + `/api/budget/month` (same pattern `SinkingFundsHelper` already uses) and displays the global `dineroAAsignar` anchored to the primary account. |
| 2026-07-09 | M10 | Sinking-fund group "Asignar" is a savings transfer, not a budget expense, when `source_account_id` is Off-Budget | GitHub #15: the group's monthly contribution (`Σ sinkingFundCalc`) still writes to `budget_allocations.assigned_amount` unchanged — that money is a transfer to the user's own Off-Budget savings account, and PRD §4.5 already treats On-Budget→Off-Budget transfers as leaving the budgeted universe. The real expense only registers when the fund is paid (`POST /api/helpers/sinking-funds/[id]/pay`, an `expense` transaction from `source_account_id`). `SinkingFundsHelper.tsx` presents this distinction explicitly ("Ahorrar este mes" vs. "Vence este mes") rather than changing what gets assigned. This assumes an Off-Budget source account — if a fund's `source_account_id` is On-Budget, the current-month due amount should be assigned directly instead; not handled by this change. |
| 2026-07-10 | M05 | CC "Pago · X" category exclusion must check `type === 'adjustment'`, never `category_id` alone | User-reported bug: a real transfer that pays off a credit card is deliberately auto-tagged with the card's "Pago · X" category (`QuickAddFormBody`'s locked `ccPaymentCategory`), so any balance-exclusion filter matching by category alone (`sumBalancesByAccount`, `net-worth-history`, `reconciliation`'s `app_balance`) was silently dropping that real payment from the source account's balance — inflating it by the sum of every card payment ever made. Fixed everywhere by requiring `type === 'adjustment'` in addition to the category match. Only the synthetic same-account mirror row is `type='adjustment'`; a real payment transfer is `type='transfer'` and is now correctly counted. |
| 2026-07-10 | M05 | Removed CC mirror `adjustment` transaction creation entirely | The stored mirror row was never actually read: `budget/month/route.ts` already recomputes "Pago · X" activity synthetically from the linked account's real expense/transfer transactions (`ccAcctMap`), not from the mirror's stored amount. Keeping it only added a transaction-list entry with no bank counterpart and enabled the exclusion bug above. `ccMirrorAmount` (now unused) and its creation code in `POST /api/transactions` and `POST /api/scheduled-transactions/[id]/confirm` were deleted; the "Pago · X" system category itself is untouched (still created on credit_card account creation, still used by the budget engine). Historical mirror rows are cleaned up by a pending migration (`20260710000001_remove_cc_mirror_transactions.sql`). |
| 2026-07-10 | M03/M08 | "Disponible para ahorrar/invertir" scopes *income* to the primary account, keeps *assigned* global, and reflects overspending same-month | User's mental model (an Excel row: "income − expected expenses of the month") needed income specific to their operating account, not summed across every on-budget account — the global `dineroAAsignar` figure could differ arbitrarily from what actually lands in their checking account when they hold multiple on-budget accounts. `budget/month/route.ts` accumulates `incomeThisMonthPrimary`/`incomeLastMonthNextFlagPrimary` (same `countsAsIncome` rule, filtered to `tx.account_id === primaryAccountId`) in the same transaction loop, and calls `computeDineroAAsignar` a second time with those scoped income figures plus the *same* global `totalAllocatedThisMonth` — the user explicitly confirmed "asignado" should stay global (all categories, not just "fixed" ones), so no new schema/flag was needed. **Unlike the global `dineroAAsignar`**, which only penalizes overspending one month later (via `negativeRolloverPrevMonth`, the *previous* month's negative Disponible), `primaryAccountAvailable` passes `negativeThisMonth` (sum of negative Disponible for the *target* month) in that same parameter slot — since `computeDisponibles` recurses month-over-month, the target month's negative Disponible already subsumes any prior unresolved deficit, so `negativeThisMonth` *replaces* rather than adds to `negativeRolloverPrevMonth` (using both would double-count). Net effect: an overspent or unbudgeted category dents this card immediately, same month — the user explicitly asked for this after realizing the assigned-only formula could overstate what's safe to move to savings mid-month. Returned as `BudgetMonthData.primaryAccountAvailable` (`null` when no account is marked primary). Still a projection, not a balance reconciliation — the number expected to match the bank exactly is the account balance in the accounts list (`sumBalancesByAccount`), not this KPI. |
| 2026-07-10 | M03 | Account list rows deep-link to `/transactions?account_id=` | Tapping an account should show its filtered history (e.g. "what did I spend and pay on this card"). `TransactionsClient.tsx` already read `account_id` from `useSearchParams()` for this exact purpose — no change needed there. `AccountCard.tsx` wraps icon+name+balance in a `<Link>` (not the edit button, which stays a sibling `<button>` to avoid nesting interactive elements inside an anchor). |
| 2026-07-10 | M09 | Dashboard Gastos/group breakdown now count negative reconciliation adjustments as expense | Audit finding: a negative `adjustment` transaction (reconciliation shortfall) already reduces the account balance and, if categorized, already reduces that category's Disponible in `/budget` (`activitiesMap` counts any categorized transaction regardless of type) — but `dashboard/route.ts`'s `total_expense`/`group_breakdown` only checked `tx.type === 'expense'`, silently excluding it and letting Dashboard and Budget disagree on "how much did I spend" for a month with a shortfall. Added a `countsAsExpense` check mirroring the existing `countsAsIncome` pattern: `tx.type === 'expense' \|\| (tx.type === 'adjustment' && amount < 0 && !isCcMirror)`. |
| 2026-07-10 | M10 | Sinking-fund rows show "Ahorrado $X de $Y (Z%)" with a progress bar | User: "no sé cuánto es el monto una vez he ahorrado... mi vista para saber cuál es mi límite." `FundRow` already received `allocated` (the fund's share of the group's saved balance, from `waterfallAllocate`) as a prop but only used it to derive the monthly-contribution figure — never rendered it. Added a progress line + thin bar (same visual pattern as `EmergencyFundHelper`'s tier bar) showing `allocated` vs. `fund.target_amount`, hidden once the fund is paid (nothing left to track). |
| 2026-07-10 | M03/M10 | `emergency-fund/route.ts` now reuses `sumBalancesByAccount` instead of summing transactions inline | Code-hygiene finding during the dashboard audit: the route recomputed account balances by hand, bypassing the CC-mirror exclusion centralized in `sumBalancesByAccount`. Not currently exploitable (a credit_card account can never be `is_emergency_fund` — the UI only exposes that toggle for off-budget accounts, and credit_card is locked to on-budget), but duplicated logic that would silently diverge if that constraint ever changes. |
