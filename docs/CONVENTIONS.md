## Validation patterns

Use Zod for all external input (form data, Route Handler bodies, URL params).

```typescript
// src/types/transaction.ts
import { z } from 'zod'

export const transactionSchema = z.object({
  date: z.string().regex(/^\d{4}-\d{2}-\d{2}$/),
  type: z.enum(['expense', 'income', 'transfer', 'adjustment', 'opening_balance']),
  account_id: z.string().uuid(),
  amount: z.number().multipleOf(0.01).positive(),
  category_id: z.string().uuid().nullable().optional(),
  payee: z.string().max(255).optional(),
  memo: z.string().max(1000).optional(),
  tags: z.array(z.string().max(50)).default([]),
  next_month: z.boolean().default(false),
})

export type TransactionInput = z.infer<typeof transactionSchema>
```

**Rules:**
- Never trust client-supplied `user_id` — always derive from `supabase.auth.getUser()` server-side.
- Validate at the Route Handler boundary; internal functions may assume valid data.
- All monetary amounts are `NUMERIC(12,2)` in DB — enforce 2 decimal places, reject more.

## Error handling

Typed Result pattern for all server operations:

```typescript
type Result<T> = { data: T; error: null } | { data: null; error: string }
```

**Route Handlers:**
```typescript
// success
return NextResponse.json({ data: result })

// failure
return NextResponse.json({ error: 'Message' }, { status: 400 })  // validation
return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
return NextResponse.json({ error: 'Internal error' }, { status: 500 })
```

**Client Components:**
- Surface errors via shadcn/ui `Sonner` toast.
- Never swallow errors silently — always `console.error` in development.
- User-facing messages must be human-readable, not raw Supabase error strings.

## Auth patterns

```typescript
// Server Component or Route Handler — verify session
import { createServerClient } from '@/lib/supabase/server'

const supabase = createServerClient()
const { data: { user } } = await supabase.auth.getUser()
if (!user) redirect('/login')
// user.id is the safe, server-derived user_id to use in all DB writes
```

```typescript
// src/middleware.ts — protects all /(app)/* routes
// Checks session cookie, redirects unauthenticated users to /login
// Also checks email_confirmed_at — unverified users redirected to /verify-email
```

```typescript
// Client Component — auth state subscription (root layout)
import { createBrowserClient } from '@/lib/supabase/client'
const supabase = createBrowserClient()
supabase.auth.onAuthStateChange((event, session) => {
  if (event === 'SIGNED_OUT') router.push('/login')
})
```

**Inactivity timeout:** 30-minute timer in `useInactivityTimer` hook.
On timeout: call `supabase.auth.signOut()` then `router.push('/login')`.

**"Recuérdame":** Pass `{ persistSession: true }` to `signInWithPassword()`.
Without it, session ends when the browser tab closes (default Supabase behavior with cookies is persistent; must set session duration to 0 for non-persistent).

**Token storage:** Access token in memory only. Refresh token in `httpOnly; Secure; SameSite=Strict` cookie — managed by `@supabase/ssr`, do not override.

## Architecture decisions

| Date | Module | Decision | Reason |
|---|---|---|---|
| 2026-06-28 | M00 | Add `display_order` to `categories` | PRD requires drag-and-drop reorder; TRD schema was missing the field — updated in TRD §4.2 |
| 2026-06-28 | All | Testing: Vitest + Testing Library + Playwright | Chosen at project init; no framework was specified in TRD |
| 2026-06-28 | M00 | `is_system` column added to `category_groups` | Required to distinguish the hidden "Sistema" system group from user groups |
| 2026-06-28 | M00 | System group has no fixed UUID — query by `is_system = true` | Cannot seed a `category_groups` row at migration time (user_id FK requires real auth user); group created per-user by `handle_new_user()` DB webhook function |
| 2026-06-28 | M01 | `user_settings` + system `category_group` created via Supabase DB Webhook | `handle_new_user()` function in migration; must be wired in Supabase dashboard; ensures atomicity and avoids race conditions |
| 2026-06-28 | M01 | Zod v4: use `.issues` not `.errors` on `ZodError` | Project uses Zod ^4.4.3; `.errors` was removed in v4 — use `result.error.issues[0].message` |
| 2026-06-28 | M01 | Middleware uses `getUser()` not `getSession()` | `getSession()` trusts the cookie without server-side verification and can be spoofed; `getUser()` validates the JWT with Supabase on every request |
| 2026-06-28 | M01 | Open redirect guard on auth callback `next` params | Validate `next` is relative (`startsWith('/')` and not `startsWith('//')`) before redirecting; external URLs fall back to `/budget` |
| 2026-06-28 | All | Next.js 16: proxy.ts not middleware.ts | Next.js 16 deprecated `middleware.ts`; use `src/proxy.ts` with `export async function proxy()` — not `middleware()` |
| 2026-06-28 | All | Run `npm install` after adding deps to package.json | Agents must run `npm install` before committing when they add packages; omitting it breaks the build for downstream worktrees |
| 2026-06-28 | M02 | Tailwind v4 class dark mode: `@custom-variant dark (&:is(.dark, .dark *))` | `tailwind.config.js` does not exist in v4; dark mode strategy must be declared in CSS via `@custom-variant` |
| 2026-06-28 | M02 | Accent theme via CSS vars `--ac/--ab/--aB/--a-shadow` + `data-accent` on `<html>` | Three accent options (azul/violeta/esmeralda); use `var(--ac)` or Tailwind `bg-[var(--ac)]` in components |
| 2026-06-28 | M02 | Theme persisted to `localStorage` (Zustand persist); DB sync deferred to M01 auth | Cannot sync to `user_settings.theme` before auth is in place; M01 must wire the sync |
| 2026-06-28 | M02 | Navigation order: Presupuesto → Cuentas → Dashboard → Helpers → Config | DESIGN.md visual reference takes precedence; PRD §5.7 listed different order |
| 2026-06-28 | M04 | Server Component + Client Component split for pages with mutations | `page.tsx` fetches initial data server-side; `*Client.tsx` handles state + mutations with optimistic updates. Avoids `react-hooks/set-state-in-effect` lint errors from `useEffect`+`setState` data fetching pattern |
| 2026-06-28 | M04 | Form sheet inner body pattern for bottom sheets | Extract stateful form UI to an inner `*FormBody` component rendered inside `<BottomSheet>`. Since BottomSheet returns null when closed, the body unmounts and mounts fresh — `useState(prop)` always initializes correctly without `useEffect` |
| 2026-06-28 | M04 | Next.js 16 dynamic route params are async | Route Handler params: `{ params }: { params: Promise<{ id: string }> }` — must `await params` before accessing id |
| 2026-06-28 | M04 | `/settings` and `/categories` added to `PROTECTED_PREFIXES` in `proxy.ts` | Both routes were unprotected — auth check was missing for settings page |
| 2026-06-28 | M04 | @dnd-kit/core + @dnd-kit/sortable for drag-and-drop | PointerSensor with `distance: 8` prevents accidental drags on button clicks. Nested DndContexts (outer for groups, inner per group for categories) — system groups excluded from sortable |
| 2026-06-29 | M05 | Transaction amount sign convention | User enters positive amount. Server applies sign: expense = negative, income = positive. Transfer: source leg = negative, dest leg = positive. CC mirror = positive. All sign logic in `src/lib/zbb/transactions.ts`. |
| 2026-06-29 | M05 | Transfer pair FK insertion order | No real DB transaction available from JS. Insert leg1 without pair_id → insert leg2 with pair_id=leg1.id → UPDATE leg1.pair_id=leg2.id. For delete: nullify both pair_ids first, then delete both rows. |
| 2026-06-29 | M05 | CC mirror adjustment is non-fatal | If the system CC category is not found, the expense is still recorded. A `console.warn` is logged. M08 budget view must handle missing mirror transactions gracefully. |
| 2026-06-29 | M05 | FAB Quick Add form: mount-on-open pattern | `QuickAddFormBody` is mounted only when the FAB is open (conditional render). It fetches accounts+categories on mount (useEffect with [] deps), then unmounts fresh on close. Same pattern as M04 form sheets. |
