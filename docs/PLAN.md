>

> - Edit amount + confirm: same but with updated amount.
> - Skip: advance `next_due_date` without inserting a transaction.
>
> **`next_due_date` advancement logic:**
> - daily: +1 day
> - weekly: +7 days
> - monthly: +1 calendar month (same day, handle month-end edge cases)
> - yearly: +1 year
>
> **No auto-insertion** (PRD ¬ß5.1.2): The app never auto-records a scheduled transaction.
> User always confirms. This is a hard product requirement.
>
> **Frequency options:** `daily | weekly | monthly | yearly` (TRD schema).

- **Status:** ‚úÖ done
- **Type:** sequential (wave 5, parallel with M06 + M07)
- **Depends on:** M05
- **Worktree:** wt-m08-budget

- **Migrations:** ‚Äî (schema in M00; `budget_template` JSONB in `user_settings`)

#### AI Notes
> **"Disponible" formula** (TRD ¬ß4.3 ‚Äî recursive):
> ```
> Disponible(M, C) = allocated(M, C) + rollover(M‚àí1, C) ‚àí activity(M, C)
> rollover(M, C) = Disponible(M, C)  [recursive; base = 0 if no prior allocation]
> activity(M, C) = SUM(transactions.amount WHERE On-Budget account, category=C, month=M)
> ```
> Compute server-side in the Route Handler. Fetch all months from the earliest `budget_months`
> record to the current month, then walk forward. Do NOT compute recursively per-row on the client.
>
> **"Dinero a Asignar" formula** (TRD ¬ß8.1):
> ```
> = SUM(income in On-Budget accounts, month M, next_month=false)
> + SUM(income month M‚àí1, next_month=true)
> ‚àí SUM(budget_allocations.assigned_amount, month M)
> + rollover_negativo(M‚àí1)  [sum of negative Disponibles from prior month]
> ```
>
> **Inline editing:** Clicking "Asignado" cell opens an input in-place. On blur/Enter, PATCH the
> `budget_allocations` record and optimistically update "Dinero a Asignar" in the Zustand store.
> Recalculate "Disponible" for that category row after the patch.
>
> **Budget template** (PRD ¬ß5.3.1): One template per user, stored as JSONB in `user_settings.budget_template`.
> "Save as Template" overwrites the existing template (one-step confirmation).
> "Apply Template" upserts `budget_allocations` for the current month using template amounts.
>
> **Month navigation:** Past months are editable. Editing a past month triggers cascading rollover
> recalculation ‚Äî implement **lazy** (recalculate when user navigates to each affected month),
> not eager (TRD ¬ß8.3). Mark affected months as "stale" in the Zustand store to trigger refetch.
>
> **Trends panel** (PRD ¬ß5.4): Side panel opened from a category row click.
> - Bar chart: last 6 months, Asignado vs. Actividad per month (Recharts).
> - KPIs: avg real spend (last 3 months), month of highest spend.
> - Lazy-load when panel opens ‚Äî do not prefetch all categories.
>
> **System categories** (`is_system = true`): Show in read-only mode in the budget table.
> No assignment input. Display the accumulated "Pago ¬∑ Visa" amount so the user knows what to pay.
>
> **`budget_months` lazy creation:** When the user navigates to a month with no `budget_months` record,
> create it on the fly. Do not pre-create all months.
>
> **Activity sign convention (M08 decision):** TRD formula `disponible = allocated + rollover ‚àí activity` uses
> "activity" as an absolute spending amount (positive = spent money). Implementation uses the signed DB sum
> (`activity = SUM(tx.amount)` ‚Äî negative for net spending) with `disponible = allocated + rollover + activity`.
> These are equivalent: `activity_abs = -activity_signed`. The UI displays raw signed activity
> (negative = net spending, which is the conventional budget display).
>
> **Allocation endpoint design:** `POST /api/budget/allocations` handles both create and update via upsert
> (`onConflict: budget_month_id,category_id`). No PATCH endpoint needed ‚Äî the client never needs the allocation ID.
>
> **Optimistic update scope:** Editing an allocation updates `assigned`, `disponible` (delta), and `dineroAAsignar`
> (-delta) in the client state. For past month edits, months after the edited month are marked stale in
> `useBudgetStore` so they refetch on navigation.
>
> **TrendsPanel remount strategy:** `key={trendsCategory}` in BudgetClient forces a fresh mount per category,
> so TrendsPanel initializes as `{ status: 'loading' }` without needing synchronous setState in the effect.
> This satisfies the `react-hooks/set-state-in-effect` lint rule.
>
> **Key files:**
> - Pure functions: `src/lib/zbb/budget.ts`
> - Tests: `src/lib/zbb/__tests__/budget.test.ts`
> - Types: `src/types/budget.ts`
> - Store: `src/stores/budget.store.ts`
> - API: `src/app/api/budget/month/route.ts`, `allocations/route.ts`, `template/route.ts`, `template/apply/route.ts`, `trends/[categoryId]/route.ts`
> - UI: `src/components/budget/` (BudgetClient, BudgetTable, MonthNavigator, DineroAAsignarKPI, TemplateActions, TrendsPanel)
