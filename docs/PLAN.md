>
> **Tables to create** (all from TRD §4.2 + one extra needed for system categories):
> `accounts`, `category_groups`, `categories`, `budget_months`, `budget_allocations`,

> **RLS pattern** — apply to every table (TRD §3.1):
> ```sql
> ALTER TABLE <table> ENABLE ROW LEVEL SECURITY;
> CREATE POLICY "user_isolation" ON <table>
>   FOR ALL
>   USING (auth.uid() = user_id)
>   WITH CHECK (auth.uid() = user_id);
> ```
>
> **EXCEPTION — `budget_allocations` has no `user_id`** — RLS must join through `budget_months`:
> ```sql
> CREATE POLICY "user_isolation" ON budget_allocations
>   FOR ALL
>   USING (
>     (SELECT user_id FROM budget_months WHERE id = budget_month_id) = auth.uid()
>   )
>   WITH CHECK (
>     (SELECT user_id FROM budget_months WHERE id = budget_month_id) = auth.uid()
>   );
> ```
>
> **Missing field from TRD schema — add `display_order` to `categories`** (PRD §5.3.2 requires drag-and-drop reorder for categories, not only groups; `category_groups` has it but `categories` does not — flagged as ambiguity):

> display_order INTEGER NOT NULL DEFAULT 0
> ```
>
> **System group:** Create a seeded `category_groups` row for the system group used by CC payment categories. Use a known UUID constant so all agents can reference it. Name: `"Sistema"`, hidden in UI.
>
> **Indexes** (TRD §6.1 — include all five):

> CREATE INDEX idx_transactions_category ON transactions(user_id, category_id, date);

- **Migrations:** — (Supabase Auth manages its own tables; `user_settings` row auto-created on first login)

#### AI Notes
> **Supabase Auth handles all password hashing** (bcrypt). Do not implement custom hashing (TRD §2.1).
>
> **Token strategy** (TRD §2.2):
> - Access token: 1-hour TTL, stored in memory only (never localStorage).
> - Refresh token: 30-day TTL, stored in `httpOnly; Secure; SameSite=Strict` cookie via `@supabase/ssr`.
>
> **Email verification flow** (TRD §2.3):
> - Create route `app/auth/confirm/route.ts` that exchanges the Supabase OTP token.
> - Middleware blocks access to `/(app)/*` until `email_confirmed_at` is set.
>
> **Password reset** (TRD §2.4):
> - `supabase.auth.resetPasswordForEmail(email)` → email with magic link.
> - `app/auth/reset-password/page.tsx` → calls `supabase.auth.updateUser({ password })`.
>
> **"Recuérdame"** — persist session flag. Without it, session should end when browser closes.
> Consult `@supabase/ssr` docs for the exact cookie persistence config — do not guess.
>
> **Inactivity timeout** — 30-minute timer (PRD §3.2). Implement as `useInactivityTimer` hook
> that resets on any user interaction event. On timeout: `supabase.auth.signOut()` + redirect to `/login`.
>
> **`user_settings` row:** Insert a default `user_settings` row for the new user inside a
> Supabase Database Webhook or an Edge Function triggered on `auth.users` insert — not in the Next.js app layer.

### M08 — Budget ZBB
- **Status:** ⏳ pending
- **Type:** sequential (wave 5, parallel with M06 + M07)
- **Depends on:** M05
- **Worktree:** —
- **web:** ⏳ Budget main view (home screen after login), month navigator, "Dinero a Asignar" KPI, category table (Asignado / Actividad / Disponible), inline assign editing, budget template (apply + save), trends panel (side panel per category — 6-month chart)
- **db:** ⏳ Route Handler for "Dinero a Asignar" computation; Route Handler for Disponible per month; Route Handler for budget allocations CRUD; Route Handler for template save/apply
- **Tests:** ⏳ Unit test rollover calculation (cascading months); unit test Dinero a Asignar formula
- **Migrations:** — (schema in M00; `budget_template` JSONB in `user_settings`)

#### AI Notes
> **"Disponible" formula** (TRD §4.3 — recursive):
> ```
> Disponible(M, C) = allocated(M, C) + rollover(M−1, C) − activity(M, C)
> rollover(M, C) = Disponible(M, C)  [recursive; base = 0 if no prior allocation]
> activity(M, C) = SUM(transactions.amount WHERE On-Budget account, category=C, month=M)
> ```
> Compute server-side in the Route Handler. Fetch all months from the earliest `budget_months`
> record to the current month, then walk forward. Do NOT compute recursively per-row on the client.
>
> **"Dinero a Asignar" formula** (TRD §8.1):
> ```
> = SUM(income in On-Budget accounts, month M, next_month=false)
> + SUM(income month M−1, next_month=true)
> − SUM(budget_allocations.assigned_amount, month M)
> + rollover_negativo(M−1)  [sum of negative Disponibles from prior month]
> ```
>
> **Inline editing:** Clicking "Asignado" cell opens an input in-place. On blur/Enter, PATCH the
> `budget_allocations` record and optimistically update "Dinero a Asignar" in the Zustand store.
> Recalculate "Disponible" for that category row after the patch.
>
> **Budget template** (PRD §5.3.1): One template per user, stored as JSONB in `user_settings.budget_template`.
> "Save as Template" overwrites the existing template (one-step confirmation).
> "Apply Template" upserts `budget_allocations` for the current month using template amounts.
>
> **Month navigation:** Past months are editable. Editing a past month triggers cascading rollover
> recalculation — implement **lazy** (recalculate when user navigates to each affected month),
> not eager (TRD §8.3). Mark affected months as "stale" in the Zustand store to trigger refetch.
>
> **Trends panel** (PRD §5.4): Side panel opened from a category row click.
> - Bar chart: last 6 months, Asignado vs. Actividad per month (Recharts).
> - KPIs: avg real spend (last 3 months), month of highest spend.
> - Lazy-load when panel opens — do not prefetch all categories.
>
> **System categories** (`is_system = true`): Show in read-only mode in the budget table.
> No assignment input. Display the accumulated "Pago · Visa" amount so the user knows what to pay.
>
> **`budget_months` lazy creation:** When the user navigates to a month with no `budget_months` record,
> create it on the fly. Do not pre-create all months.
